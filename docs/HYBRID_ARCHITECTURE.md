# Harmonia Hybrid Architecture (GDScript + C#)

## Objective

Use Godot 4.x with a deliberate hybrid model:

- GDScript for scene orchestration and fast gameplay iteration.
- C# for compute-heavy and reusable domain/service modules.

## Language Responsibility Matrix

### Use GDScript For

- Scene scripts, Node lifecycle hooks, and UI controllers.
- Signal wiring between scenes and singleton managers.
- Rapid gameplay rules iteration and prototype logic.
- Lightweight adapters that call into C# services.

### Use C# For

- Pitch analysis math pipelines that become performance bottlenecks.
- Data transformation, scoring engines, and reusable battle calculators.
- Save validation/serialization helpers with strict typed models.
- Tooling and editor-side utilities where C# libraries offer better leverage.

### Keep At Manager Boundaries

- GameStateManager: state transitions and high-level flow.
- AudioProcessor: audio capture coordination and note output contracts.
- BattleManager: target-note checks, grading, HP/score updates.
- LocalDataManager: persistence API, schema validation, storage adapters.

## Cross-Language Design Rules

- Scene and UI layers should not contain heavy business logic.
- Manager APIs must stay stable and typed.
- Keep payload contracts explicit and versionable.
- Prefer event/signal handoffs over direct deep object reach-through.
- Isolate platform/runtime-specific code behind manager/service interfaces.

## Suggested Project Layout

- scripts/gd/autoloads/: singleton GDScript manager entrypoints.
- scripts/gd/ui/: scene/UI scripts and signal bindings.
- scripts/csharp/domain/: core C# services and pure logic modules.
- scripts/csharp/infrastructure/: persistence and adapter implementations.
- scripts/shared/contracts/: schema constants and payload keys used by both sides.

## Main Configuration Checklist

### Godot/.NET Setup

1. Use Godot 4 .NET build for this project.
2. Ensure C# support is available in the active IDE/runtime.
3. Keep project opening in the same Godot variant used to build C# assemblies.
4. Commit C# project files generated by Godot (.csproj and solution files) once initialized.

### Autoload and Script Setup

1. Register manager autoloads in project settings (GameStateManager, AudioProcessor, BattleManager, LocalDataManager).
2. Keep autoload public API language-agnostic (typed method contracts and payload dictionaries/objects).
3. Route scene-level calls through managers, not directly into deep C# internals.

### Audio and DSP Setup

1. Keep Record bus isolated and muted on monitor path to avoid feedback loops.
2. Use confidence/noise-floor thresholds before publishing detected note events.
3. Run heavier analysis in C# only if profiling proves GDScript is insufficient.

### Build and Source Control

1. Keep one source of truth for constants/contracts to avoid drift.
2. Add repeatable build verification for both GDScript parse and C# compile steps.
3. Use the same .NET SDK major version across contributors.

## Bare-Truth Tradeoffs

- GDScript-only is faster to iterate but can bottleneck in heavy numeric paths.
- C# improves structure/performance for complex logic but increases setup and tooling complexity.
- Hybrid is strongest when boundaries are strict; it fails when logic is duplicated across both languages.

## Initial Split Recommendation for Harmonia

- Start AudioProcessor orchestration and UI glue in GDScript.
- Move pitch core math and scoring formula engines to C# once baseline behavior is validated.
- Keep BattleManager and LocalDataManager APIs stable so internals can move between languages safely.
